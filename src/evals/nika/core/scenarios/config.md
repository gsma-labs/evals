# Scenarios – How to Add and Register a New Scenario

This document explains how to **construct a new scenario** under `build_images/scenarios` and how to **register it** in `scenario.yaml` so it can be used for dataset generation and evaluation.

## Directory Layout

Scenarios are grouped by **category** (e.g. `data_center`, `campus`, `isp`, `sdn`, `p4`). Each scenario is a folder that provides one or more runnable labs (topologies).

```
build_images/scenarios/
├── config.md              # This file – how to add scenarios
├── scenario.yaml          # Registry: all scenarios and their compose paths
├── data_center/
│   ├── dc_clos_service/   # Scenario: DC with DNS/Web services
│   │   ├── config.md      # Topology, protocol, testing (per-scenario doc)
│   │   ├── lab_generator.py
│   │   ├── size_s/
│   │   │   ├── compose.yaml
│   │   │   └── topology/
│   │   │       └── lab.conf
│   │   ├── size_m/ ...
│   │   └── size_l/ ...
│   └── dc_clos_worker/    # Scenario: BGP-only DC
│       ├── config.md
│       ├── lab_generator.py
│       └── size_s/ size_m/ size_l/ ...
├── campus/
│   ├── campus_static/
│   │   └── config.md
│   └── campus_dhcp/
│       └── config.md
├── isp/
├── sdn/
└── p4/
```

- **Category** = first level (`data_center`, `campus`, `isp`, `sdn`, `p4`).
- **Scenario** = named folder under a category (e.g. `dc_clos_service`, `campus_dhcp`).
- **Variant** = optional size or flavor. Many scenarios use `size_s`, `size_m`, `size_l`; P4 scenarios often have a single variant (no `size_*`).

## How to Construct a Scenario

### 1. Choose category and name

- Create `build_images/scenarios/<category>/<scenario_name>/` (e.g. `isp/my_rip_lab/`).

### 2. Provide a runnable lab

Each runnable lab is a **compose file** that Kathara can start. You can get it in one of two ways:

**Option A – Pre-built compose**

- Put `compose.yaml` (and optionally `topology/lab.conf`) under a variant folder, e.g. `size_s/compose.yaml`, `size_s/topology/lab.conf`.
- Paths in `scenario.yaml` will point to that `compose.yaml`.

**Option B – Generated by script (recommended for parameterized topologies)**

- Add a `lab_generator.py` in the scenario root.
- The script should write `lab.conf` (and any `.startup` or config files) under a variant directory, e.g. `size_s/topology/` or `size_m/topology/`.
- After generation, `compose.yaml` is produced from `lab.conf` (e.g. via `inspect-kathara`’s `generate_compose_for_inspect`). The dataset loader can run the generator and then generate compose; for the registry we list the **resulting** `compose.yaml` path (e.g. `isp/my_rip_lab/size_s/compose.yaml`).

### 3. Lab naming

- In `topology/lab.conf`, set `LAB_NAME="..."`. This name is used as the Kathara lab name and should be unique enough for logs and debugging.
- Use a short, readable name (e.g. `rip_small_internet_vpn`, `dc_clos_worker`, `dc_clos_service`, `campus_dhcp`).

### 4. Write `config.md` for the scenario

- In the **scenario folder** (e.g. `data_center/dc_clos_service/config.md`), add a `config.md` that describes:
  - **Overview**: what the topology is and what it is used for.
  - **Topology / connections**: diagram (ASCII) and how devices are connected.
  - **Protocols**: e.g. OSPF, BGP, RIP, WireGuard, SDN controller, P4 pipeline.
  - **Addressing**: IP scheme, VLANs if any.
  - **Services**: DNS, HTTP, DHCP, etc., if present.
  - **Directory structure**: where `lab.conf`, `compose.yaml`, startup scripts, and configs live.
  - **Testing commands**: how to verify connectivity and protocols (e.g. `vtysh`, `ping`, `curl`).

Use existing scenario `config.md` files as templates (e.g. `campus/config.md`, `data_center/dc_clos_service/config.md`).

## How to Register in scenario.yaml

After the scenario is built and at least one `compose.yaml` exists (per variant), add one **registry entry per runnable lab** (i.e. per compose file) in `build_images/scenarios/scenario.yaml`.

### Fields

| Field | Required | Description |
|-------|----------|-------------|
| `id` | Yes | Unique ID, e.g. `data_center/dc_clos_service/size_s` or `p4/p4_counter`. |
| `category` | Yes | One of: `data_center`, `campus`, `isp`, `sdn`, `p4`. |
| `name` | Yes | Scenario name (folder name), e.g. `dc_clos_service`, `campus_dhcp`. |
| `size` | Yes | `s`, `m`, `l`, or `null` if the scenario has no size variants. |
| `compose_path` | Yes | Path to `compose.yaml` **relative to** `build_images/scenarios`. Example: `data_center/dc_clos_service/size_s/compose.yaml`. |
| `lab_name` | Yes | Value of `LAB_NAME` in the lab’s `lab.conf` (e.g. `dc_clos_service`, `rip_small_internet_vpn`). |
| `lab_conf_path` | Optional | Path to that lab’s `lab.conf` relative to `build_images/scenarios`. |
| `has_lab_generator` | Optional | `true` if the scenario has a `lab_generator.py` that produces this variant. |

### Example: one variant (no size)

```yaml
- id: p4/p4_counter
  category: p4
  name: p4_counter
  size: null
  compose_path: p4/p4_counter/compose.yaml
  lab_name: p4_counter
  lab_conf_path: p4/p4_counter/topology/lab.conf
  has_lab_generator: true
```

### Example: three sizes

```yaml
- id: campus/campus_dhcp/size_s
  category: campus
  name: campus_dhcp
  size: s
  compose_path: campus/campus_dhcp/size_s/compose.yaml
  lab_name: campus_dhcp
  lab_conf_path: campus/campus_dhcp/size_s/topology/lab.conf
  has_lab_generator: true
# ... size_m and size_l entries
```

### Path convention

- All paths in `scenario.yaml` are **relative to** `build_images/scenarios`.
- In `build_images/datasets/dataset.yaml`, the sandbox field uses the same relative path: `sandbox: [kathara, "<compose_path>"]`, e.g. `[kathara, "data_center/dc_clos_service/size_s/compose.yaml"]`.

## Checklist for a New Scenario

1. Create `build_images/scenarios/<category>/<scenario_name>/`.
2. Implement the lab (compose + optional `lab_generator.py` and `topology/lab.conf`).
3. Set `LAB_NAME` in `lab.conf` and ensure at least one `compose.yaml` exists per variant.
4. Add a **scenario-level** `config.md` (topology, protocols, addressing, testing).
5. Append one or more entries to `build_images/scenarios/scenario.yaml` (one per compose/variant).
6. Optionally add dataset samples in `build_images/datasets/dataset.yaml` that reference the new `compose_path`.

## Reference: scenario.yaml

The full list of registered scenarios and their compose paths is in:

**`build_images/scenarios/scenario.yaml`**

Each scenario’s details (topology, protocols, testing) are in:

**`build_images/scenarios/<category>/<scenario_name>/config.md`**
